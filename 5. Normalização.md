# 1. Normalização

> A normalização é um conjunto de regras utilizado para organizar as tabelas de um banco de dados.

- Reduz redundância de dados.
- Aumenta a integridade das informações.
- Facilita a manutenção e a consistência do banco de dados.
- Existem seis formas normais, porém as três primeiras são as mais utilizadas.
- Uma tabela só pode avançar para a próxima forma normal quando atender às regras da anterior.

---
# 2. Primeira Forma Normal (1FN)

> Elimina grupos repetitivos e garante que cada coluna possua apenas valores atômicos (indivisíveis).

- Cada coluna deve armazenar apenas um único valor por registro.
- Não podem existir listas, arrays ou múltiplos valores em uma mesma célula.
- Cada linha representa um único registro.
- Cada coluna representa apenas um atributo.
- Toda tabela deve possuir uma chave primária.
- É a base para as demais formas normais.

## Como identificar

- Uma coluna armazena vários valores.
- Existem colunas repetidas (`telefone1`, `telefone2`, `telefone3`).
- Uma célula contém uma lista separada por vírgulas.

### Exemplo

❌ Incorreto

| id | nome | telefones |
|----|------|-----------|
| 1 | João | 9999, 8888 |

✅ Correto

**CLIENTE**

| id | nome |
|----|------|
| 1 | João |

**TELEFONE**

| id_cliente | telefone |
|------------|-----------|
| 1 | 9999 |
| 1 | 8888 |

---

# 3. Segunda Forma Normal (2FN)

> Elimina as dependências parciais, garantindo que todo atributo não chave dependa da chave primária inteira.

- A tabela deve estar obrigatoriamente na Primeira Forma Normal (1FN).
- Aplica-se principalmente quando existe chave primária composta.
- Todos os atributos não chave devem depender da chave completa.
- Se um atributo depender apenas de parte da chave, existe uma dependência parcial.
- Geralmente a solução é dividir a tabela em novas tabelas.

## Dependência Funcional

> Ocorre  quando o valor de um atributo determina o valor de outro atributo.

### Exemplo

```
CPF → Nome
```

## Dependência Parcial

> O atributo depende de apenas parte da chave primária composta.

### Exemplo

```
(id_pedido, id_produto)

id_produto → nome_produto
```

## Dependência Total

> O atributo depende da chave primária composta completa.

### Exemplo

```
(id_pedido, id_produto)

(id_pedido, id_produto) → quantidade
```

## Como identificar

- A chave primária é composta.
- Um atributo não chave depende apenas de um dos campos da chave.
- Existem informações repetidas por causa dessa dependência parcial.

### Exemplo

| id_pedido | id_produto | nome_produto | quantidade |
| --------- | ---------- | ------------ | ---------- |
| 1         | 10         | Mouse        | 2          |

```
id_produto → nome_produto      (Dependência Parcial)

(id_pedido, id_produto) → quantidade      (Dependência Total)
```

# 4. Terceira Forma Normal (3FN)

> Elimina as **dependências transitivas**. Um atributo não chave deve depender **diretamente da chave primária**, e não de outro atributo não chave.

- A tabela deve estar obrigatoriamente na Primeira Forma Normal (1FN) e na Segunda Forma Normal (2FN).
    
- Nenhum atributo não chave deve depender de outro atributo não chave.
    
- Todos os atributos não chave devem depender **diretamente da chave primária**.
    
- Quando uma dependência transitiva é encontrada, os atributos envolvidos devem ser separados em uma nova entidade.

## 4.1 Dependência transitiva

Uma dependência transitiva ocorre quando:

> Chave primária → atributo não chave → outro atributo não chave

Ou seja, um atributo depende da chave primária **indiretamente**, por meio de outro atributo.

## Exemplo

Digamos que temos uma entidade `Disciplina` com os seguintes atributos:

```sql
disciplina_id
descricao
carga_horaria
codigo_professor
nome_professor
email_professor
fk_idcurso
```

Temos as seguintes dependências:

- `descricao`, `carga_horaria` e `codigo_professor` dependem de `disciplina_id`.
    
- Porém, `nome_professor` e `email_professor` dependem de `codigo_professor`.
    

Portanto:

```text
disciplina_id → codigo_professor → nome_professor, email_professor
```

Isso é uma **dependência transitiva**, pois `nome_professor` e `email_professor` não dependem diretamente da chave primária `disciplina_id`. Eles dependem de `codigo_professor`, que é um atributo não chave na entidade `Disciplina`.

## Como resolver?

1. Identificar o atributo **determinante**:
    
    - `codigo_professor`
        
2. Identificar os atributos que dependem dele:
    
    - `nome_professor`
        
    - `email_professor`
        
3. Criar uma nova entidade chamada `Professor`.
    
4. Usar `codigo_professor` como **chave primária** de `Professor`.
    
5. Manter `codigo_professor` em `Disciplina` como **chave estrangeira**, criando o relacionamento entre `Disciplina` e `Professor`.
