# 1. Introdução

> Relacionamentos são associações entre tabelas de um banco de dados relacional. 
> Eles permitem conectar registros por meio de chaves, garantindo organização e [[9. Integridade Dos Dados|integridade]] dos dados.

- O modelo relacional organiza os dados em **tabelas (relações)**.
- As tabelas podem se relacionar entre si por meio de **chaves estrangeiras (Foreign Keys)**.
- Os relacionamentos permitem associar registros de uma tabela com registros de outra.
- Todo relacionamento possui uma **cardinalidade**, representada por um par de valores.
- Cardinalidade define **quantas ocorrências de uma entidade podem estar relacionadas com outra**.
	- Evitar problemas como:
		- duplicação de informações;
		- redundância;
		- inconsistência;
		- dificuldade para atualizar dados;
			- perda da integridade (conformidade referencial).

---

# 2. Relacionamento 1:N (Um para Muitos)

> É um relacionamento onde um registro de uma tabela pode estar associado a vários registros de outra tabela, enquanto cada registro da segunda tabela pertence a apenas um registro da primeira.

- Um registro da tabela principal pode estar relacionado com **vários registros** da tabela secundária.
- Cada registro da tabela secundária pertence a **apenas um** registro da tabela principal.
- Utiliza **chave estrangeira (Foreign Key)** para representar o relacionamento.
- A cardinalidade informa tanto a quantidade quanto a obrigatoriedade do relacionamento.
    - **1** → obrigatório.
    - **0..N** → opcional e pode possuir vários registros.
- É recomendado utilizar **constraints**, pois elas garantem a integridade dos dados.
    - Quando a constraint é criada dentro do `CREATE TABLE`, normalmente o banco gera um nome automaticamente.
    - Podemos definir ou alterar o nome posteriormente utilizando `ALTER TABLE`.
    - Bons nomes para constraints facilitam a manutenção e a documentação (Dicionário de Dados).

### Exemplo

> Um funcionário possui vários telefones.

- O funcionário pode existir sem possuir telefone.
- Um telefone obrigatoriamente pertence a um funcionário.

# 3. Relacionamento 1:1 (Um para Um)

> É um relacionamento onde cada registro de uma tabela está associado a apenas um registro da outra tabela.

- Geralmente é utilizado para separar informações em tabelas diferentes, evitando tabelas muito grandes ou agrupando dados por responsabilidade.
- A chave estrangeira normalmente recebe uma restrição **UNIQUE**, garantindo que não existam múltiplos relacionamentos.

### Exemplo

> Um funcionário possui um endereço.

- Em vez de adicionar diversas colunas de endereço na tabela `Funcionário`, cria-se uma tabela `Endereço`.

# 4. Relacionamento N:N (Muitos para Muitos)

> É um relacionamento onde vários registros de uma tabela podem estar associados a vários registros de outra. Para representá-lo, é necessária uma tabela auxiliar.

- Um registro da primeira tabela pode estar relacionado com vários registros da segunda.
- Um registro da segunda tabela pode estar relacionado com vários registros da primeira.
- Esse relacionamento **não pode ser implementado diretamente** apenas com chaves estrangeiras.
- É necessário criar uma **tabela auxiliar (tabela associativa ou tabela de junção)**.
- A tabela auxiliar armazena as chaves primárias das duas tabelas como chaves estrangeiras.

### Por que existe uma tabela auxiliar?

> Ela resolve o relacionamento muitos para muitos transformando-o em dois relacionamentos 1:N.

- Uma única chave estrangeira não consegue representar múltiplos registros dos dois lados.
- Cada registro da tabela auxiliar representa uma associação entre os registros das tabelas relacionadas.
- Tem a responsabilidade de armazenar a relação das entidades.

### Exemplo
Um aluno pode cursar várias disciplinas.
Uma disciplina pode possuir vários alunos.

A tabela auxiliar poderia ser:

- `Aluno_Disciplina`
    - `id_aluno`
    - `id_disciplina`

# 5. Tipos de Relacionamento

>Os relacionamentos são classificados de acordo com a quantidade de entidades envolvidas.

Os principais tipos são:

- **Unário (recursivo)**
    
- **Binário**
    
- **Ternário**
    

Existem outros tipos, como quaternário, quinário etc., porém são menos comuns, pois relacionamentos com muitas entidades geralmente podem ser decompostos em relacionamentos menores.

---

# 5.1 Relacionamento Unário

## Conceito

> Um relacionamento unário ocorre quando uma entidade se relaciona com ela mesma.

- Também é chamado de **relacionamento recursivo**, pois a tabela possui uma referência para ela própria.

- É utilizado quando existe uma relação entre registros da mesma entidade.

- Envolve apenas **uma entidade**.
    
- A tabela possui uma chave estrangeira apontando para ela mesma.
    
- Normalmente representa hierarquias ou dependências.
    

### Exemplos: Funcionário supervisiona funcionário

Um funcionário pode supervisionar outro funcionário.

```
FUNCIONÁRIO
------------
id_funcionario
nome
id_supervisor (FK)
```

Relacionamento:

> Funcionário **supervisiona** Funcionário.

Exemplo de dados:

|id|nome|id_supervisor|
|---|---|---|
|1|João|NULL|
|2|Maria|1|
|3|Pedro|1|

Nesse caso:

- João supervisiona Maria.
    
- João supervisiona Pedro.
    

---

# 5.2 Relacionamento Binário

> Um relacionamento binário ocorre quando duas entidades diferentes se relacionam.

- É o tipo de relacionamento mais comum em bancos relacionais.

-  Envolve **duas entidades**.
    
- Normalmente é representado através de uma chave estrangeira.
    
- Pode possuir diferentes cardinalidades:
    
    - 1:1 (um para um)
        
    - 1:N (um para muitos)
        
    - N:N (muitos para muitos)
        

---

## Exemplos: Cliente realiza Pedido

Entidades:

```
CLIENTE
--------
id_cliente
nome


PEDIDO
--------
id_pedido
data
id_cliente (FK)
```

Relacionamento:

> Cliente **realiza** Pedido.

Um cliente pode realizar vários pedidos.

Representação:

```
CLIENTE 1 -------- N PEDIDO
```

---

### Aluno matricula-se em Curso

Nesse caso existe uma relação muitos para muitos.

Um aluno pode fazer vários cursos e um curso pode possuir vários alunos.

```
ALUNO
------
id_aluno


CURSO
------
id_curso


MATRICULA
----------
id_aluno (FK)
id_curso (FK)
```

Relacionamento:

> Aluno **matricula-se** em Curso.

A tabela intermediária transforma o relacionamento N:N em dois relacionamentos 1:N.

---

# 5.3 Relacionamento Ternário

## Conceito

> Um relacionamento ternário ocorre quando três entidades participam de um mesmo relacionamento.

É utilizado quando a associação depende da combinação das três entidades.

### Características

- Envolve **três entidades simultaneamente**.
    
- Geralmente cria uma tabela associativa.
    
- Não deve ser dividido simplesmente quando a informação depende das três entidades juntas.
    

---

## Exemplo: Professor ministra Disciplina em Turma

Entidades:

```
PROFESSOR
----------
id_professor


DISCIPLINA
-----------
id_disciplina


TURMA
------
id_turma
```

Relacionamento:

> Professor **ministra** Disciplina em Turma.

Tabela criada:

```
MINISTRA
---------
id_professor (FK)
id_disciplina (FK)
id_turma (FK)
```

Exemplo:

|Professor|Disciplina|Turma|
|---|---|---|
|João|Banco de Dados|3A|
|Maria|Redes|2B|

A informação só faz sentido considerando as três entidades juntas.

# Outros Tipos

## Conceito

> Quando um relacionamento envolve mais entidades, ele recebe uma classificação baseada na quantidade de entidades participantes.

Exemplos:

|Tipo|Quantidade de entidades|
|---|---|
|Unário|1 entidade|
|Binário|2 entidades|
|Ternário|3 entidades|
|Quaternário|4 entidades|
|Quinário|5 entidades|
|Sextário|6 entidades|

### Observação

> Quanto maior a quantidade de entidades em um relacionamento, maior a complexidade da modelagem.

Por isso, relacionamentos com muitas entidades devem ser analisados para verificar se podem ser separados em relacionamentos menores.

# 6. Nomeação dos Relacionamentos

## Conceito

> O nome de um relacionamento deve ser representado por um verbo no presente do indicativo, indicando uma ação ou associação entre as entidades.

- O objetivo é tornar a modelagem mais fácil de compreender, inclusive por pessoas que não possuem conhecimento técnico em banco de dados.

- Um bom nome de relacionamento deve:

	- Ser um **verbo no presente do indicativo**.
	    
	- Representar uma ação clara.
	    
	- Evitar nomes genéricos.
	    
	- Facilitar a leitura do modelo.
    

---

## Exemplos corretos

### Cliente e Pedido

```
CLIENTE -------- PEDIDO
```

Relacionamento:

> Cliente **realiza** Pedido.

---

### Funcionário e Empresa

```
FUNCIONÁRIO -------- EMPRESA
```

Relacionamento:

> Funcionário **trabalha** em Empresa.

---

### Médico e Paciente

```
MÉDICO -------- PACIENTE
```

Relacionamento:

> Médico **atende** Paciente.
