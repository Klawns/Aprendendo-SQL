# 1. PK Composta

> Chave primária formada pela combinação de duas ou mais colunas para identificar um registro de forma única.

- Muito utilizada em relacionamentos **N:N (Muitos para Muitos)**.
- Normalmente é composta pelas **chaves estrangeiras (FKs)** da tabela associativa.
- Garante que o mesmo relacionamento não seja cadastrado duas vezes.
- Evita a necessidade de criar uma coluna de identificação apenas para a tabela intermediária.

### Exemplo

Tabela `Aluno_Disciplina`

- `id_aluno` (FK)
- `id_disciplina` (FK)

```sql
PRIMARY KEY (id_aluno, id_disciplina)
```

Cada combinação de aluno e disciplina pode existir apenas uma vez.

### Quando utilizar

- Quando a tabela existe apenas para representar um relacionamento.
- Quando não há necessidade de referenciar a tabela intermediária em outros relacionamentos.
- Quando a combinação das FKs já identifica unicamente cada registro.

# 2. Surrogate Key × Natural Key

## 2.1 Chaves

> Chaves são atributos (ou conjunto de atributos) utilizados para identificar unicamente um registro em uma tabela.

- Identificam uma linha de forma exclusiva.
- Também são utilizadas para estabelecer [[7. Relacionamentos|Relacionamentos]] entre tabelas.
- Podem ser naturais ou artificiais (Surrogate Keys).
## 2.2 Natural Key

> Chave baseada em uma informação existente no mundo real.

- Possui significado para o negócio.
- Pode ser utilizada como chave primária.
- Pode sofrer alterações conforme a regra de negócio.
-  Exemplos:
	- CPF
	- Placa de veículo
	- E-mail
	- Matrícula
## 2.3 Surrogate Key

> Chave artificial criada pelo sistema exclusivamente para identificar um registro.

- Não possui significado para o negócio.
- Geralmente é um número inteiro (`INT`, `BIGINT`) ou um `UUID`.
- É utilizada como chave primária.
- Normalmente é gerada automaticamente pelo banco de dados ou pela aplicação.
- Não deve ser manipulada pelo usuário.
- Não depende das regras de negócio.
- Alterações nos dados da entidade não afetam sua identificação.
 - Vantagens:
	- Mantém a chave primária pequena e simples.
	- Facilita os relacionamentos entre tabelas.
	- Melhora o desempenho dos índices quando comparada a chaves compostas muito grandes.
	- Mudanças nas regras de negócio não exigem alteração da chave primária.

## 2.4 Surrogate Key em tabelas associativas

> Uma tabela associativa pode utilizar tanto uma **PK composta** quanto uma **Surrogate Key**. A escolha depende da necessidade da modelagem.

### Opção 1 — PK Composta

A própria combinação das FKs identifica o relacionamento.

```text
VOCE_CELULAR

id_voce
id_celular

PRIMARY KEY (id_voce, id_celular)
```
- Vantagens:
	- Não cria uma coluna adicional.
	- Estrutura mais simples.
	- Garante que não existam relacionamentos duplicados.
	- É a abordagem mais comum quando a tabela apenas representa o relacionamento.
### Opção 2 — Surrogate Key

Cria-se uma coluna exclusiva para identificar cada relacionamento.

```text
VOCE_CELULAR

id_voce_celular
id_voce
id_celular
```

```sql
PRIMARY KEY (id_voce_celular)

UNIQUE (id_voce, id_celular)
```

Nesse caso:

- `id_voce_celular` é a **Surrogate Key** da tabela.
- `id_voce` e `id_celular` continuam sendo FKs.
- A constraint `UNIQUE` impede que o mesmo relacionamento seja cadastrado duas vezes.
### Quando utilizar

É recomendada quando:
- Outras tabelas precisarão referenciar a tabela associativa.
- O relacionamento possui atributos próprios.

### Exemplo

```text
Aluno_Disciplina

id
id_aluno
id_disciplina
data_matricula
nota
status
```

Nesse cenário, a tabela deixa de representar apenas um relacionamento e passa a ser uma entidade do sistema.

> **Observação:** utilizar uma Surrogate Key não elimina a necessidade de impedir registros duplicados. Por isso, normalmente também é criada uma constraint `UNIQUE (id_voce, id_celular)`.
