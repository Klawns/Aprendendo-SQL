# 1. Introdução

> Relacionamentos são associações entre tabelas de um banco de dados relacional. Eles permitem conectar registros por meio de chaves, garantindo organização e integridade dos dados.

- O modelo relacional organiza os dados em **tabelas (relações)**.
- As tabelas podem se relacionar entre si por meio de **chaves estrangeiras (Foreign Keys)**.
- Os relacionamentos permitem associar registros de uma tabela com registros de outra.
- Todo relacionamento possui uma **cardinalidade**, representada por um par de valores.
- Cardinalidade define **quantas ocorrências de uma entidade podem estar relacionadas com outra**.
	- Evitar problemas como:
		- duplicação de informações;
		- redundância;
		- inconsistência;
		- dificuldade para atualizar dados;
			- perda da integridade (conformidade referencial).

# 2. Relacionamento 1:N (Um para Muitos)

> É um relacionamento onde um registro de uma tabela pode estar associado a vários registros de outra tabela, enquanto cada registro da segunda tabela pertence a apenas um registro da primeira.

- Um registro da tabela principal pode estar relacionado com **vários registros** da tabela secundária.
- Cada registro da tabela secundária pertence a **apenas um** registro da tabela principal.
- Utiliza **chave estrangeira (Foreign Key)** para representar o relacionamento.
- A cardinalidade informa tanto a quantidade quanto a obrigatoriedade do relacionamento.
    - **1** → obrigatório.
    - **0..N** → opcional e pode possuir vários registros.
- É recomendado utilizar **constraints**, pois elas garantem a integridade dos dados.
    - Quando a constraint é criada dentro do `CREATE TABLE`, normalmente o banco gera um nome automaticamente.
    - Podemos definir ou alterar o nome posteriormente utilizando `ALTER TABLE`.
    - Bons nomes para constraints facilitam a manutenção e a documentação (Dicionário de Dados).

### Exemplo

> Um funcionário possui vários telefones.

- O funcionário pode existir sem possuir telefone.
- Um telefone obrigatoriamente pertence a um funcionário.

# 3. Relacionamento 1:1 (Um para Um)

> É um relacionamento onde cada registro de uma tabela está associado a apenas um registro da outra tabela.

- Geralmente é utilizado para separar informações em tabelas diferentes, evitando tabelas muito grandes ou agrupando dados por responsabilidade.
- A chave estrangeira normalmente recebe uma restrição **UNIQUE**, garantindo que não existam múltiplos relacionamentos.

### Exemplo

> Um funcionário possui um endereço.

- Em vez de adicionar diversas colunas de endereço na tabela `Funcionário`, cria-se uma tabela `Endereço`.

# 4. Relacionamento N:N (Muitos para Muitos)

> É um relacionamento onde vários registros de uma tabela podem estar associados a vários registros de outra. Para representá-lo, é necessária uma tabela auxiliar.

- Um registro da primeira tabela pode estar relacionado com vários registros da segunda.
- Um registro da segunda tabela pode estar relacionado com vários registros da primeira.
- Esse relacionamento **não pode ser implementado diretamente** apenas com chaves estrangeiras.
- É necessário criar uma **tabela auxiliar (tabela associativa ou tabela de junção)**.
- A tabela auxiliar armazena as chaves primárias das duas tabelas como chaves estrangeiras.

### Por que existe uma tabela auxiliar?

> Ela resolve o relacionamento muitos para muitos transformando-o em dois relacionamentos 1:N.

- Uma única chave estrangeira não consegue representar múltiplos registros dos dois lados.
- Cada registro da tabela auxiliar representa uma associação entre os registros das tabelas relacionadas.
- Tem a responsabilidade de armazenar a relação das entidades.

### Exemplo
Um aluno pode cursar várias disciplinas.
Uma disciplina pode possuir vários alunos.

A tabela auxiliar poderia ser:

- `Aluno_Disciplina`
    - `id_aluno`
    - `id_disciplina`

# 5. PK Composta

> Chave primária formada pela combinação de duas ou mais colunas para identificar um registro de forma única.

- Muito utilizada em relacionamentos **N:N (Muitos para Muitos)**.
- Normalmente é composta pelas **chaves estrangeiras (FKs)** da tabela associativa.
- Garante que o mesmo relacionamento não seja cadastrado duas vezes.
- Evita a necessidade de criar uma coluna de identificação apenas para a tabela intermediária.

### Exemplo

Tabela `Aluno_Disciplina`

- `id_aluno` (FK)
- `id_disciplina` (FK)

```sql
PRIMARY KEY (id_aluno, id_disciplina)
```

Cada combinação de aluno e disciplina pode existir apenas uma vez.

### Quando utilizar

- Quando a tabela existe apenas para representar um relacionamento.
- Quando não há necessidade de referenciar a tabela intermediária em outros relacionamentos.
- Quando a combinação das FKs já identifica unicamente cada registro.

# 6. Surrogate Key × Natural Key

## 6.1 Chaves

> Chaves são atributos (ou conjunto de atributos) utilizados para identificar unicamente um registro em uma tabela.

- Identificam uma linha de forma exclusiva.
- Também são utilizadas para estabelecer relacionamentos entre tabelas.
- Podem ser naturais ou artificiais (Surrogate Keys).
## 6.2 Natural Key

> Chave baseada em uma informação existente no mundo real.

- Possui significado para o negócio.
- Pode ser utilizada como chave primária.
- Pode sofrer alterações conforme a regra de negócio.
-  Exemplos:
	- CPF
	- Placa de veículo
	- E-mail
	- Matrícula
## 6.3 Surrogate Key

> Chave artificial criada pelo sistema exclusivamente para identificar um registro.

- Não possui significado para o negócio.
- Geralmente é um número inteiro (`INT`, `BIGINT`) ou um `UUID`.
- É utilizada como chave primária.
- Normalmente é gerada automaticamente pelo banco de dados ou pela aplicação.
- Não deve ser manipulada pelo usuário.
- Não depende das regras de negócio.
- Alterações nos dados da entidade não afetam sua identificação.
 - Vantagens:
	- Mantém a chave primária pequena e simples.
	- Facilita os relacionamentos entre tabelas.
	- Melhora o desempenho dos índices quando comparada a chaves compostas muito grandes.
	- Mudanças nas regras de negócio não exigem alteração da chave primária.

## 6.4 Surrogate Key em tabelas associativas

> Uma tabela associativa pode utilizar tanto uma **PK composta** quanto uma **Surrogate Key**. A escolha depende da necessidade da modelagem.

### Opção 1 — PK Composta

A própria combinação das FKs identifica o relacionamento.

```text
VOCE_CELULAR

id_voce
id_celular

PRIMARY KEY (id_voce, id_celular)
```
- Vantagens:
	- Não cria uma coluna adicional.
	- Estrutura mais simples.
	- Garante que não existam relacionamentos duplicados.
	- É a abordagem mais comum quando a tabela apenas representa o relacionamento.
### Opção 2 — Surrogate Key

Cria-se uma coluna exclusiva para identificar cada relacionamento.

```text
VOCE_CELULAR

id_voce_celular
id_voce
id_celular
```

```sql
PRIMARY KEY (id_voce_celular)

UNIQUE (id_voce, id_celular)
```

Nesse caso:

- `id_voce_celular` é a **Surrogate Key** da tabela.
- `id_voce` e `id_celular` continuam sendo FKs.
- A constraint `UNIQUE` impede que o mesmo relacionamento seja cadastrado duas vezes.
### Quando utilizar

É recomendada quando:
- Outras tabelas precisarão referenciar a tabela associativa.
- O relacionamento possui atributos próprios.

### Exemplo

```text
Aluno_Disciplina

id
id_aluno
id_disciplina
data_matricula
nota
status
```

Nesse cenário, a tabela deixa de representar apenas um relacionamento e passa a ser uma entidade do sistema.

> **Observação:** utilizar uma Surrogate Key não elimina a necessidade de impedir registros duplicados. Por isso, normalmente também é criada uma constraint `UNIQUE (id_voce, id_celular)`.

# 7. Constraints

> Constraints são regras impostas pelo banco de dados para garantir a integridade e a consistência das informações armazenadas.

Elas permitem definir:

- Se uma coluna aceita valores nulos.
- Se uma coluna deve possuir valores únicos.
- Se uma coluna representa uma chave primária.
- Se uma coluna representa uma chave estrangeira.
- Regras de validação dos dados.

## Principais Constraints

- `PRIMARY KEY` → identifica unicamente um registro.
- `FOREIGN KEY` → cria relacionamento entre tabelas.
- `UNIQUE` → impede valores duplicados.
- `NOT NULL` → impede valores nulos.
- `CHECK` → valida uma condição.
- `DEFAULT` → define um valor padrão.

## Nomeando Constraints

> É recomendado nomear as constraints para facilitar manutenção e documentação do banco de dados.

Quando uma constraint é criada dentro do `CREATE TABLE`, o banco pode gerar um nome automaticamente.

Exemplo:

```sql
CONSTRAINT PK_CLIENTE PRIMARY KEY (id_cliente)

CONSTRAINT FK_PEDIDO_CLIENTE
FOREIGN KEY (id_cliente)
REFERENCES Cliente(id_cliente)
```

# 8. Integridade Referencial

> Garante a consistência e a precisão dos relacionamentos entre tabelas.

- Uma chave estrangeira somente pode apontar para uma chave primária existente.
- Evita registros órfãos e referências inválidas.
- É garantida pelas constraints `FOREIGN KEY`.
- `RESTRICT`: Impede excluir ou alterar um registro que possua registros dependentes.
- `CASCADE`: Ao excluir ou alterar o registro principal, os registros relacionados também são automaticamente excluídos ou atualizados.
- `SET NULL`: Ao excluir o registro principal, a chave estrangeira dos registros dependentes recebe `NULL`.
- `SET DEFAULT`: Ao excluir o registro principal, a chave estrangeira recebe seu valor padrão (quando suportado pelo SGBD).
- `NO ACTION`: Comportamento semelhante ao `RESTRICT` na maioria dos bancos, impedindo a operação caso existam registros relacionados.
